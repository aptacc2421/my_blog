<a id="intro"></a>

## Finals Week and a "Simple" Feature

Mid-June 2026, finals week. But adding `/proc` RSS accounting to StarryOS had been on the backlog for a while—`RssAnon`, `RssFile`, and `VmRSS` in `/proc/[pid]/status`, plus the seven fields in `/proc/[pid]/statm`. The scope was clear, Linux already defined the semantics, and I figured I could wrap it up in a day or two.

I needed tests. I did not have time to read the kernel line by line—exams ran through the following week. So I opened Cursor and gave one instruction: write a proc memory monitor test suite, referencing Linux mainline source, the design philosophy of Linux kselftest, and how user-space tools like psutil/top actually use `/proc`.

Cursor complied. It produced a C program with 12 test functions covering single-process mmap, fork COW, mremap, mprotect, and similar scenarios. The 12 tests were numbered in execution order; item 4 verified file-mapped COW page faults, and item 6 verified parent-child COW RSS increments—the one that ultimately failed. The test design followed the reference frame I asked for: read `/proc/self/status`, parse fields, assert with `TEST PASSED` / `FAIL:`. The core helper was `read_self_status_kb`—open `/proc/self/status`, find the named field, return the value in KB.

There was a detail I did not notice at the time. Looking back, it decided the next ~22 hours.

`read_self_status_kb` was implemented with `fopen`—open `/proc/self/status`, `fgets` a line, parse the RSS value, `fclose`. Three lines, seemingly harmless.

But StarryOS uses musl libc, and in the musl build StarryOS ships with, `fopen` triggers an anonymous `mmap` to allocate the FILE buffer. In StarryOS kernel RSS accounting, that memory is counted toward `RssAnon`.

In other words, **the act of measuring RSS changed RSS**—measurement perturbation. On the morning of June 13, I did not know that yet.

> **Editor's note**: The "three AIs" in the title are **Cursor** (generated the initial test draft), **DeepSeek** (Claude CLI—ran code debugging and log analysis), and **GPT** (ChatGPT web—framed the investigation and strategy). Below, "I" performed many technical steps—adding kernel logs, reading source, inserting tracepoints, analyzing data—but in practice DeepSeek did those under my direction. My role in this investigation was closer to **messenger** and **decision-maker**: forwarding GPT's strategy to DeepSeek and DeepSeek's results back to GPT. The narrative keeps the "I" voice for readability, but readers should know: the AI actually wrote code, read logs, and ran experiments. See the section "The Real Structural Problem" later.


<a id="phenomenon"></a>

## The Phenomenon

The test ran under QEMU loongarch64 (single-core / smp=1). After 60 seconds, QEMU timed out. Kernel log timestamps advanced only to 2.505 seconds, then nothing new appeared.

Digging into the logs, three numbers jumped out:

**138 → 95**. Between two forks, the parent process's charge count dropped from 138 to 95—43 entries gone. In between, only mremap and mprotect ran, which should reclaim at most a few pages. A drop of 43 entries was clearly abnormal—**later proved to be log lines from different processes mixed together, not the same counter falling; see "Dead End #2".**

**"already Anon, syncing" kept repeating**. The same virtual address `0x601c2000` triggered `cow_file_write_to_anon` over and over, each time reporting "this page is already Anon." By design, the first COW write on a MAP_PRIVATE file mapping reclassifies one page from File to Anon; after that, this path should not run again. Repeated entry meant page classification and the charge map might have drifted apart.—**Later confirmed: the parent had already done mprotect + COW write on that address before fork; File→Anon conversion finished on the parent side. The child inherited an already-Anon page, so COW naturally reported "already Anon." Kernel behavior was correct—normal semantics, unrelated to any bug.**

**The kernel hung silently**. The last meaningful kernel activity was `handle_cow_fault(VA:0x3ffffe000, count=2)` on the child's stack page. No panic, no oops, no user-space output. The test binary died before reaching any assertion.

All three phenomena pointed the same way: an RSS accounting charge map consistency problem. Charge leak explained 138→95; classification drift explained repeated "already Anon"; eventually some COW path underflowed into panic, with the panic message never making it to the serial port.

That judgment was not random. It explained all three anomalies at once. It pointed at the most complex code in the kernel—MemoryAccounting, which I had just written the week before. It matched my intuition for what a "kernel bug should look like."

The problem: one premise was skipped in that judgment—**does the bug actually exist?**


<a id="dead-end-1"></a>

## Dead End #1: Deadlock—But the Kernel Never Died

My first investigation angle was "where did it hang?" Logs showed fork #3's PRE-FORK, but POST-CLONE never appeared. That meant `clone_map` was stuck while walking VMAs.

DeepSeek offered a natural explanation: on SMP, FRAME_TABLE lock and AddrSpace Mutex deadlocked in AB-BA order—a textbook multicore bug pattern. I accepted that direction, so DeepSeek read the full `clone_map` implementation, traced FRAME_TABLE lock acquisition order, and imagined a complex scenario: CPU A holds AddrSpace Mutex in `clone_map` and waits on FRAME_TABLE; CPU B holds FRAME_TABLE on the page-fault path and waits on AddrSpace Mutex.

To verify, I added `CLONE-PROGRESS` logs at every key step in `clone_map`, recompiled, reran Docker, waited for QEMU again.

CLONE-PROGRESS said: **`clone_map` executed every line normally.** Every VMA was walked, every frame copied. No blocking. No deadlock.

In hindsight, AB-BA deadlock was theoretically possible on smp=1+sched-rr (preemption)—via preemption rather than simultaneous lock hold—but the odds were tiny. The real issue was a slow serial port: each `[RSS_VERIFY]` line is ~120 bytes; thousands of lines × ~0.7 ms per line on the wire = kernel time at 2.5 seconds of logs needing 60+ seconds wall clock to finish printing. "POST-CLONE never appeared" did not mean it never ran—it meant it never finished transmitting.

Lesson #1: In QEMU kernel debugging, missing log output ≠ code did not run. The serial port is a slow link; when log generation outpaces transmission, output queues up—absence of a line does not mean absence of execution.


<a id="dead-end-2"></a>

## Dead End #2: The 138→95 "Charge Leak"

After raising the timeout to 120s, the test finally printed the real failure:

```
FAIL: child COW write increases RssAnon
parent_rss_anon=92 KB  (= 23 pages)
child_rss_anon=92 KB   (= 23 pages)
expected=96 KB          (= 24 pages)
```

After the child's COW write, RssAnon should be 24 pages (23 inherited from fork + 1 from COW), but it was only 23.

Alongside that failure sat the 138→95 charge drop. Both clues pointed at one hypothesis: **the charge map was leaking**. `clone_map` dropped entries when copying charges, or `reconcile_fork_charges` computed wrong.

I started verifying BTreeMap internal consistency. Added duplicate detection in `record_charge`. Added missing-key detection in `remove_charge`. Added before/after length assertions in `snapshot_resident_charges`. Added two global AtomicU64 counters: `RECORD_OK` for successful inserts, `RECORD_DUP` for duplicate inserts.

Wait for Docker build. Wait for QEMU. Wait for the log stream.

`RECORD_DUP = 0`. **No duplicate insert in any test run. No missing-key remove. `mapped_pages == charge_entries` held at every fork boundary. `len_ok` was true—the BTreeMap was never modified concurrently during iteration.**

The BTreeMap was clean. The charge map was not leaking. So what was 138→95?

Only later did I understand: 138 and 95 were not two measurements of the same thing. They came from different pids at different times. My AtomicU64 counters mixed init, shell, test binary, child process—all processes' reconciliation output in one stream. 138 was process A's peak at time Ta; 95 was process B's state at time Tb. The gap was not "lost entries"—it was "unrelated observations."

Lesson #2: Global counters without PID context are unreliable diagnostic tools in a multi-process kernel. 138 and 95 looked like a variable falling from high to low, but they were never values of the same variable.


<a id="dead-end-3-5"></a>

## Dead Ends #3, #4, #5: sync Overwrite, Fork Inheritance, Cross-Process Sharing

These three fit together because they share one pattern: each found a perfect alibi in the kernel; each verification round made the kernel look more correct.

**sync_rss_atomics_from_charges overwrite hypothesis**: This function has a load-compare-store sequence. On SMP, if atomics change on another CPU between compare and store, the correct value might be overwritten wrongly. After adding an `SYNC-OVERWRITE` counter—**0**. On every call, `atomics_before == snapshot == atomics_after`. sync was always a no-op—defensive code that never actually overwrote. —**In hindsight, the system had only one CPU; the premise "another CPU modified concurrently" did not exist. During the investigation, everyone forgot that.**

**Fork inheritance hypothesis**: Suspected `clone_map` missed a VMA or VA when copying the charge map. After adding `FORK-INHERIT` logs on the fork path—**parent_anon = 22, child_anon = 22**. Inheritance was correct at every fork boundary. Parent and child had separate `acct_ptr` (different pointers), not shared.

**Cross-process BTreeMap sharing hypothesis**: Suspected parent and child shared one BTreeMap (via Arc), so the child's COW mutated the parent's stats. Verified `acct_ptr` per pid—all unique. Each process had its own MemoryAccounting object.

Three strikes against the kernel. The kernel was scrubbed clean inch by inch. BTreeMap correct, atomics correct, fork correct, isolation correct. By then I had spent roughly 8 hours ruling out every plausible failure path in kernel accounting (through root-cause identification; DeepSeek platform stats showed ~260M tokens consumed cumulative, input context plus model output—mostly from long logs repeatedly filling the context window).

Then I made the fatal synthesis: **kernel accounting is all correct + the 23→22 drop is real = some undiscovered kernel mechanism must be deleting charges.** That reasoning was internally consistent—if the BTreeMap decrease was not sync/fork/sharing bugs, it had to be some subtler kernel operation off the verified paths.

Its only flaw was a wrong premise.


<a id="turning-point"></a>

## Turning Point: Admitting 23→22 Was Real

After every accounting module checked out, one fact could not be dodged: **between T1 (**`/proc/self/status` **read) and T2 (fork), the parent process BTreeMap anon count really went from 23 to 22.**

I added two log points in `stats.rs`. `SNAPSHOT-T1` prints when `ProcessMemStats::collect()` is triggered by a `/proc` read. `SNAPSHOT-T2` prints on the fork path the same way. Both log pid, acct_ptr, anon count, file count, and nanosecond timestamps.

The logs came back:

```
[SNAPSHOT-T1] pid=26 acct_ptr=0x9000000001b92070 anon=23 file=74 ts=2995115000
[SNAPSHOT-T2] pid=26 acct_ptr=0x9000000001b92070 anon=22 file=75 ts=3180000000
```

Same `acct_ptr`. Same `snapshot_resident_charges()` call. 185 ms between reads. The anon count in the BTreeMap really dropped by one page. Not statistical noise, not an SMP phantom read—a BTreeMap entry was deleted.

Between the two reads, user space only did `pipe()`—two synchronous syscalls. No allocation, no free, no VMA change.

How could the BTreeMap change with the Mutex held by a single thread and no syscalls in between? That became the core contradiction of the whole investigation.


<a id="dead-end-6"></a>

## Dead End #6: Delayed Cleanup—the Longest Wrong Road

After ruling out every "at the same instant" explanation, only "time" remained. The BTreeMap change did not happen inside the T1–T2 window—it was **delayed from an earlier moment and only became visible in this window**.

The natural candidate: test #4 (`test_map_private_file_write_first_fault`) did `munmap` at VA:0xf1000. That was an Anon charge—`munmap` calls `remove_charge(Anon)` and deletes one Anon entry from the BTreeMap. If that delete was not promptly "committed" (on SMP, cache line not invalidated), test #6's `/proc` read (on another CPU) might see stale state—the Anon charge still in the BTreeMap.

The hypothesis was internally airtight. It explained why the BTreeMap changed in a single-threaded, no-syscall T1–T2 window—the mutation did not happen in those 185 ms; it happened earlier, with delayed visibility on this CPU.

I fed this hypothesis into the investigation pipeline. Looking back, almost every response made me more confident in that direction:

> "This is neither a test logic error nor a MemoryAccounting calculation error. It is a timing issue between `munmap`'s BTreeMap mutation and the subsequent `/proc` read."

The conclusion sounded professional and rigorous—separating "test design," "data structure correctness," and "execution timing." It precisely ruled out my main suspicion—test code. How could a few hundred lines of C be more suspect than tens of thousands of lines of Rust kernel?

To test the hypothesis, I ran two experiments. Like every round in this investigation—GPT proposed, I forwarded, DeepSeek (Claude CLI) implemented.

First experiment: add a `caller` parameter to `remove_charge`, pass `caller=sys_munmap` from `sys_munmap`. Also add tracepoints at `sys_mmap`, `sys_munmap`, and `sys_fork` entry, logging pid, VA, and nanosecond timestamps.

Second experiment: wrap each `read_self_status_kb` call in the test C code with `clock_gettime`, trying to align kernel and user-space timestamps.

Logs returned. VA:0xf1000's `remove_charge` timestamp was ~2.17 s. T1 was ~2.69 s. T2 was ~3.18 s.

**VA:0xf1000 was already cleaned up during test #4.** It was not between T1 and T2. The delayed-cleanup hypothesis was falsified by timestamps.

The Anon entry that vanished between T1 and T2 was not VA:0xf1000. Then who was it?


<a id="va-16000"></a>

## VA:0x16000

During test #6 (the failing one), another `remove_charge` event appeared in the logs:

```
[CHARGE-MUTATION] ts=32501686752 pid=23 op=remove va=VA:0x16000 old_kind=Anon caller=sys_munmap
```

VA:0x16000. Not the address of any known test VMA.

I searched SYSCALL-MMAP logs. In the same time window, the mmap record for this address:

```
[SYSCALL-MMAP] ts=2697269616 pid=23 va=VA:0x16000 len=0x1000 file=false flags=READ|WRITE|USER
```

`file=false`. This was an **anonymous mapping**, not the test's MAP_PRIVATE file mapping (VA:0x15000, `file=true`). It was created before test #6's T1 measurement and destroyed after T1, lifetime ~1.73 ms.

The full causal chain:

```
2696539024 ns: test mmap(VA:0x15000, file=true)          ← file mapping
2696854544 ns: record_charge File at VA:0x15000             ← page fault fill
2697269616 ns: SYSCALL-MMAP va=0x16000 file=false           ← fopen buffer!
2697432688 ns: record_charge Anon at VA:0x16000             ← anon: 22→23
2697500000 ns: SNAPSHOT-T1 anon=23                          ← T1 snapshot (polluted value)
2699002320 ns: SYSCALL-MUNMAP va=0x16000                    ← fclose release
2699263520 ns: remove_charge Anon at VA:0x16000             ← anon: 23→22
```

At T1 snapshot anon=23; the Anon page at VA:0x16000 still existed; ~1.7 ms later it was munmap'd. The snapshot captured it; fork did not.

**One ironic detail**: The `SYSCALL-MMAP`/`SYSCALL-MUNMAP` tracepoints that captured this timeline were added to chase VA:0xf1000 delayed cleanup—a wrong direction. Instrumentation planted for the wrong reason accidentally caught the real culprit.

> Note: This timeline is from the loongarch instrumented run (baseline 22→23, 92 KB / 23 pages). The same bug reproduced on aarch64—baseline 18→19, same ±1 delta. Absolute baselines differ by architecture; the ±1 delta is identical—measurement self-perturbation of RSS is reproducible across architectures.

This anonymous mapping was not created by the test. It was created by `fopen("/proc/self/status", "r")`.

The verification chain—replayed on Linux with the same musl version afterward—strace captured identical behavior: `fopen` → `mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS)`, `fclose` → `munmap(4096)`. Two independent measurements confirmed allocation scale: LD_PRELOAD hooking `malloc` measured musl `fopen`'s total allocation request at 1264 bytes (= FILE struct + UNGET + BUFSIZ ≈ 232 + 8 + 1024); pure I/O buffer request was 1181 bytes, page-aligned to 4096—both consistent: `fopen` triggers one anonymous mmap. StarryOS had matching QEMU logs: `SYSCALL-MMAP va=0x16000 len=0x1000 file=false flags=READ|WRITE|USER`—anonymous mapping, timed exactly with the `/proc` read. Kernel paths verified: `cow.rs:123-131` marks new pages from `file=None` mappings as `RssKind::Anon`; `accounting.rs:107-108` counts them in `rss_anon`.

So the truth was:

**Pollution happened on the parent's measurement, not the child's.**

The parent called `read_self_status_kb("RssAnon")` to read its own `/proc/self/status`. On musl libc, `fopen` triggered one anonymous `mmap(4096, MAP_ANONYMOUS)`—FILE struct, unget buffer, read/write buffer totaling 1264 bytes, rounded up to 4096 by page granularity. The first write to the FILE struct triggered a page fault; StarryOS counted that anonymous page in RssAnon. Then `fgets` parsed `"RssAnon: XX kB"`—**the parent read a value one page higher than its true RSS**. That page was `fopen`'s own FILE buffer.

`fclose` closed the file; `munmap` freed the anonymous page; then `free` reclaimed the FILE struct. The parent's RSS returned to the true value (22 pages).

Then `fork`. By then the fopen buffer was long gone; the child inherited the parent's true value—22 pages, clean. Child COW write raised anon from 22 to 23. The child read its own `/proc/self/status`—23, correct.

The assertion checks `child >= parent + 1`. parent used T1's polluted 23 (true 22 + fopen's 1); child was 23 (true 22 + COW's 1). 23 >= 23+1 → FAIL.

**The parent was over-counted by fopen by one page; the child was clean. Pollution on the parent, not the child.**

The bug was not in the kernel. It was in the test code. In the `fopen` line.

`fopen` → `open`. That was the fix.


<a id="test-acquittal"></a>

## The Test Was Formally Tried Twice, Acquitted Twice

During the investigation, "could it be the test?" came up twice—but not from me; the AIs raised it. GPT listed it as a category in structured investigation (Q5: "Is this a test logic problem or a kernel problem?"); DeepSeek received the question, analyzed, and acquitted. I was the messenger between the two AIs—pasting GPT's Q5 into the CLI, pasting DeepSeek's verdict back to ChatGPT.

Both times, DeepSeek delivered clear, logically tight acquittals. GPT drove the second round's delayed-cleanup investigation—caller= tracing, A/B/C classification—but the actual acquittal language came from the CLI side.

First time, DS's summary before saving a checkpoint:

> "This is neither a test logic error nor a MemoryAccounting calculation error. It is a timing issue between `munmap`'s BTreeMap mutation and the subsequent `/proc` read."

Note the structure: it denies "test logic error" and "accounting calculation error," but attributes the problem to a third bucket—"timing issue." The taxonomy looked precise but missed the only real answer: **measurement tool side effects**. `fopen`'s FILE buffer polluting RSS—this is not "test logic" (the file-read logic was correct), not "accounting calculation" (the kernel math was correct), and not a "timing issue" at all (`fopen`'s side effect exists at every timing).

The analysis used a seemingly exhaustive three-way split that excluded the actual culprit.

Second time, while investigating the "delayed cleanup" hypothesis:

> "Not a test design problem (A): test logic is correct—read /proc/self/status → fork → child COW write → read child state. No concurrency, no race assumptions. Test code triggers no unmap between the two pipes."

> "Not a MemoryAccounting problem (B): BTreeMap insert/remove/iterate operations are correct… the issue is 'counting happened at the wrong time'…"

> "It is a process lifecycle problem (C): AddrSpace::clear() timing is decoupled from scheduling…"

Precise analysis, complete reasoning, clear taxonomy. Completely wrong.

Both verdicts shared one premise: some kernel operation (munmap, AddrSpace cleanup) was delayed into the T1–T2 window. Neither analysis ever considered "the read operation itself might allocate memory"—not because of a logical hole, but because the measurement tool (`fopen`) sat outside the scope of premise checking.

The test was acquitted in advance.

But a harder question: **did the culprit's trail appear in the logs?**

Yes—at least three times.

First, early in the investigation (round 3). DS walked all VMAs looking for broken invariants. The anonymous mmap log for VA:0x16000 was right there—`file=false`, timed inside test #6's `/proc` read window. Amid the noise of 138→95, CPU26/27, and "already Anon," an anonymous mapping that looked like libc internal allocation was not worth mentioning.

Second, at the "oscillation" discovery (round 18). Same process, between two `/proc` reads, RssAnon bounced between 23 and 22. DS had already pinned the BTreeMap change between `/proc` read and fork—**one follow-up question, "who allocated memory between the two reads," would have ended at fopen.** But the question was "which kernel module deleted a page," not "what in user space added then removed a page."

Third, round 23. DS found the same VA mmap'd then munmap'd within 1 ms in the logs and proposed stack tracing. That VA was 0x16000. Attention immediately went to "process exit cleanup" and "shared MemAccounting pointer"—more complex kernel stories.

Three times, fopen's footprints were in the logs. Three times, the investigation walked past them.

This was not poor eyesight. **The problem frame decided what counted as evidence.** When the frame was "the kernel has a bug," libc allocation behavior was not in evidence space—it was "environment noise." To see fopen, you did not need more logs; you needed to switch the frame from "where is the kernel wrong" to "who changed this number."


<a id="structure-problem"></a>

## The Real Structural Problem: Not "AI Isn't Good Enough," but "The Human Was the Messenger"

Looking back at this pipeline, there is a structural issue deeper than "verify why instead of verify whether": **what did I actually do in this investigation?**

Finals week. I had Cursor generate the test. When it failed, I fed logs to Claude CLI (DeepSeek model). DeepSeek output analysis; I copy-pasted to ChatGPT web. ChatGPT output the next analysis; I copy-pasted back to DeepSeek. Across 59 conversation rounds my actual role was **message queue**—receive one model's output, forward to another.

That structure had three consequences, each worse than "the model made a mistake."

**First, errors hardened into "facts" in transit.** In round 3 DeepSeek said "CPU 26 and CPU 27 both participated in that fault." I pasted it verbatim into round 4's prompt as "confirmed fact #3." ChatGPT reasoned from that "fact." The log format was actually `{cpu_id}:{tid}`; DeepSeek misread task_id 26 as CPU 26—on smp=1, only CPU 0 exists. But I did not read code; I only read model text. Model said CPU26, I wrote CPU26. **The error was not model-only—it was "confirmed" into the next round's input at my step, with no verification from me.**

**Second, hypotheses that should not exist lived for hours.** AB-BA deadlock on sched-rr preemption on a single core was theoretically possible (via preemption, not simultaneous locks) but extremely unlikely—and the real issue was serial port bottleneck. SMP atomics overwrite, cache coherence—these were impossible on single core; there was no other CPU. The QEMU config directory name literally said "smp1"—single core. **Nobody looked at code and config during the whole investigation.** Models read code and logs; the human read model text. smp=1 was invisible from model text alone—not because it hid in some obscure config—the directory name was right there—but because nobody looked, and models did not put it in reasoning preconditions.

There is a deeper pattern: **LLMs lose "obvious facts" in long-context reasoning.** smp=1 was mentioned once at the start; 300 rounds later it was no longer active context. AI does not remember "we only have one core" like a human does—unless reminded repeatedly. This suggests AI-assisted debugging may need an auto-maintained "known constraints list" checked before each inference.

**Third, the human step added no new information.** Ideal AI-assisted debugging: human understands the system, human proposes hypotheses, AI runs verification, human judges results. What happened: AI proposed hypotheses, AI ran verification, AI analyzed results, human did Ctrl+C / Ctrl+V. Both models shared the same unexamined premise, and the person in the middle—the only entity that could step outside text and challenge premises—because exams left no time to read code, because "simple feature, quick fix," because not one assertion was independently checked outside model output—**added no new information to the pipeline.** **AI is good at enumerating possibilities; humans are good at ruling out impossibilities.** Here enumeration ran wild; exclusion never happened.

Fair credit: I was not completely hands-off. When GPT said "compare on Linux first," I compiled and ran the test on WSL and saw `FAIL: touch one page increases RSS`. But that single independent check was not to verify whether DeepSeek or GPT's claims held—it was to see if Linux behaved like StarryOS. I verified the wrong question. It did not challenge the premise—it made me think "even Linux fails; the test is just hard," pushing the test further away.

This was not an AI capability problem. DeepSeek and ChatGPT performed well enough—they read code, wrote tracepoints, analyzed logs, designed experiments. The problem was **they shared a wrong premise**, and **the one person who could break that closed loop was not inside the loop.**


<a id="premise"></a>

## Why This Premise Survived

In hindsight, four mechanisms kept the premise "the bug is in the kernel" from being tested.

**First, the test was written referencing Linux/kselftest.** It was born with authority—it mimicked Linux kselftest structure (`TEST PASSED`/`FAIL:` format, `/proc` field parsing, parent-child checks after fork), mimicked psutil/top usage. When a test is generated under "reference Linux mainline + kselftest + psutil," its credibility is swapped for the original authors' authority.

But it missed one key semantic: using `fopen` inside the process under test to read `/proc/self/status` means the read itself allocates memory counted in RSS. On glibc in my test environment, the same `fopen` did not reproduce +1 page (glibc `fopen` often extends heap via `brk`, reusing pages without a new anonymous `mmap`), but on StarryOS + musl every `fopen` independently `mmap`'s one anonymous page. The test copied Linux test *shape*, not this musl-specific side effect.

**Second, trust ranking followed complexity, not evidence.** The kernel is tens of thousands of lines of Rust across MM, VMA, COW, fork, SMP—I had written it in under a week. The test was a few hundred lines of C, simple logic, single file, dozens of calls. In the intuition sort of "which is more likely wrong," the kernel ranked above the test by roughly an order of magnitude—not because any evidence showed a kernel bug, but because the kernel had more places that *looked* like they could fail.

**Third, the trust chain's starting point never entered verification.** I verified BTreeMap insert/remove consistency. Verified atomics vs snapshot consistency. Verified fork inheritance per VA. Verified COW File→Anon reclassification. Even verified SMP Mutex release-acquire visibility. But I never verified "`read_self_status_kb`'s return equals the kernel's actual state"—because it was the measurement baseline, and baselines are out of scope. I was not asking "does the bug exist?" I was asking "why does the bug exist?"

**Fourth, pseudo-independence.** The test was "tried" at least twice by two models, each time professionally acquitted, each time spared for some "more plausible kernel explanation." The two reviews looked independent—different times, contexts, analysis paths—but shared the same unchallenged premise. A test that survives two independent reviews looks more credible than one—but both reviews were derivatives of one premise, not true independent verification.

Every verification that the kernel module was correct made me more convinced subconsciously that "there must be a kernel bug I haven't found," not "maybe the kernel is fine."

Right outcome, wrong direction.


<a id="interlude"></a>

## Interlude: Three AIs Relay-Wrote This Bug

Counting back, three AIs helped create this bug.

**Cursor autocomplete** casually wrote `fopen`—not knowing what it was writing, not knowing musl vs glibc, not knowing `fopen` would inflate RSS on the target system. It saw a `/proc` path in context and completed a seemingly reasonable file read.

**DeepSeek (Claude CLI)** wrote the rest of the test without questioning `fopen`. But in debugging it inferred fopen buffer behavior from tracepoint output—`file=false`, anonymous mapping, timestamp spanning T1. Same fact (`fopen` mmap's), unseen when writing code, seen when debugging bugs. Same model, two cognitive modes, never connected.

**GPT (ChatGPT Web)** designed the Audit Mode Q1–Q5 framework, enforcing "no guessing" discipline. It structured the investigation—classify hypotheses, rule out falsified paths, demand caller= tracing—but only saw fopen indirectly from DS output around R45. Good at organizing investigation, weak at discovering concrete facts.

No AI was omniscient. Each wrote blind in its local context. Cursor did not know musl; DS did not connect fopen side effects while writing code; GPT never read logs directly. Three AI outputs were puzzle pieces that together formed the full bug—but the person assembling the puzzle was not AI; it was the human copy-pasting during finals week.


<a id="fix"></a>

## The Fix

```diff
- FILE *fp = fopen(path, "r");
+ int fd = open(path, O_RDONLY);
```

Five functions, three lines each. Rebuild, rerun tests. Pass.

Not charge leak, not deadlock, not atomics overwrite, not fork inheritance bug, not delayed cleanup, not cache coherence. Six hypotheses falsified one by one. One line of `fopen`.


<a id="what-if"></a>

## What If…

If after every `fopen` read of `/proc/self/status`, the test immediately did a second independent read with `open`/`read`/`close` (no `fopen`, no `FILE*`), the two RSS values should match—mismatch means measurement perturbation. That check inside `read_self_status_kb` is one line of code.

If before fork the parent printed current RSS to stderr and compared with what the child inherited after fork—the fopen buffer would already be freed; parent and child baselines should match. Mismatch means a problem.

If every `/proc` read had RSS snapshots before and after, a non-zero diff means "the read itself allocated or freed memory"—no kernel code, no tracepoints, no 260M tokens.

**Not hindsight.** These three checks total under 20 lines of C—basic self-checks for any measurement system—"calibration" in physics, "sanity check" in software testing. Cursor would not add them by default because AI assumes earlier steps are correct and only the final FAIL is the bug. What actually happened: the error was planted at step one; every later assertion built on a wrong foundation and could not succeed.


<a id="ending"></a>

## Closing

This investigation ran from the first afternoon through the second—about 22 hours total. I slept; the AI did not. Actual "debugging" time—reading code, checking man pages, running experiments, comparing results—was under two hours. The rest waited on Docker builds, QEMU boot, model context windows swallowing logs, and one person's analyses copy-pasted into another model.

I am not blaming the models for mistakes. That is not the point. The point is: **when the human's only contribution in a debug loop is copy-paste, errors are not corrected—they are forwarded.**

The feature is fixed. Early-exam subjects are done. But how much of every "why right" and "why wrong" on this pipeline did I verify with my own hands?

During the investigation: zero. All code reading, log analysis, hypothesis, experiment design, and conclusions were done by AI. I copy-pasted in the middle and independently verified not one assertion.

Afterward, in retrospective: one—`fopen` to `open`, 12 tests green. Even that was confirmed only after DeepSeek wrote the fix, ran tests, and reported "passed." I did not write the fix myself.

RSS never lied. It faithfully recorded every memory change—including measurement behavior: `fopen`'s FILE buffer, COW File→Anon reclassification, the child's stack page fault. What lied was our default assumption that measurement has no side effects, and the never-verified premise: "the bug must be in the kernel."

But this investigation left more than one lesson:

1. **Constraints first.** When designing functions and tests, rule out illegal paths up front. Each step proves itself. Assertions must verify not only whether the result is right but whether intermediate state is valid.
2. **Human reviews AI for impossibility.** When AI proposes a hypothesis, ask "which known constraints does this violate?" not "is there evidence for this?"

   Specific to this investigation:
   - Deadlock hypothesis (dead end #1) → ask: what is smp? → **smp=1, one CPU.** Deadlock on preempted single core is theoretically possible but extremely unlikely. Check serial port before deadlock.
   - Cross-process sharing hypothesis (dead end #5) → ask: do parent and child share an address space after fork? → **No.** Child has its own page tables; child BTreeMap ops cannot pollute parent stats. VA isolation kills it.
   - Delayed cleanup hypothesis (dead end #6) → ask: does cache coherence delay exist on single core? → **No.** Writes on the same CPU are immediately visible to that CPU. No second CPU can "see stale."

   AI enumerates possibilities; humans rule out impossibilities. smp=1, fork semantics, VA isolation—these are not details; they are fast knives for killing bad hypotheses. Three cuts would have removed at least four of six dead ends. Nobody asked these questions when AI proposed hypotheses.
3. **Check man before code.** Specs answer "how it should be"; code answers "how it actually is." AI prefers code (grep in repo), but specs often answer "should" faster. musl `fopen` implementation—`mmap` for FILE buffer—one `strace` line shows it.
4. **Debug branches record conclusions, not edits.** Commit messages document what was falsified or confirmed, not what changed. Conclusions at each stage are the only durable output. Strengthen with `Confirms:` or `Refutes:` tags per commit.
5. **Suspect libc first on ±1.** RSS off by one page—dynamic linker faults, `fopen` FILE buffer, `malloc` brk. All more common than kernel bugs. On the 4 KB suspect list, libc ranks first.
6. **Watch AI context decay.** LLMs drop "obvious facts" in long context. smp=1 was said once at the start; 300 rounds later it was forgotten. AI-assisted debugging needs an auto-maintained "known constraints list" checked before each inference—not relying on the model to remember.
