## 期末考与一个「简单」功能

2026 年 6 月中旬，期末考周。但给 StarryOS 加 `/proc` RSS accounting 这件事已经拖了一阵——`/proc/[pid]/status` 的 `RssAnon`、`RssFile`、`VmRSS`，`/proc/[pid]/statm` 的七个字段。功能边界清晰，Linux 有现成语义，我估计一两天能收工。

我需要测试。但我没有时间逐行读内核代码——考试日程排到了下周。于是打开了 Cursor，给出了一条指令：参考 Linux 主线源码、Linux kselftest 的设计思想、以及 psutil/top 这类用户态工具的实际使用模式，写一套 proc memory monitor 测试套件。

Cursor 照做了。产出了一份 12 个测试函数的 C 程序，覆盖单进程 mmap、fork COW、mremap、mprotect 等场景。12 个测试按执行顺序编号，其中第 4 项为文件映射 COW 缺页验证，第 6 项为父子进程 COW RSS 增量验证——也是最终失败的用例。测试的设计完全模仿了我要求的参照系：读 `/proc/self/status`、解析字段、用 `TEST PASSED` / `FAIL:` 做断言。核心工具函数是 `read_self_status_kb` ——打开 `/proc/self/status`，找到指定字段，返回以 KB 为单位的值。

这里面有一个细节，我当时没有注意到。后来回头看，它决定了接下来近 22 个小时的命运。

`read_self_status_kb` 是用 `fopen` 实现的——打开 `/proc/self/status`，`fgets` 读一行，解析出 RSS 值，`fclose` 关闭。区区三行，看起来无害。

但 StarryOS 使用 musl libc，而在当前 StarryOS 所使用的 musl 实现中，`fopen` 会触发一次匿名 `mmap` 来分配 FILE 缓冲区。在 StarryOS 内核的 RSS 统计中，这块内存会被计入 `RssAnon`。

也就是说，**测量 RSS 的操作本身改变了 RSS**——测量扰动（measurement perturbation）。但在 6 月 13 日早上，我还不知道这件事。

> **叙述说明**：标题中的"三个 AI"分别是 **Cursor**（生成测试初稿）、**DeepSeek**（Claude CLI，执行代码调试与日志分析）、**GPT**（ChatGPT 网页版，负责调查框架与策略方向）。下文中"我"执行了大量技术操作——加内核日志、读源码、插 tracepoint、分析数据——实际上这些均由 DeepSeek 在我的指令下执行。我在这次调查中的角色更接近"信使"和"决策者"——把 GPT 的策略转发给 DeepSeek，把 DeepSeek 的结果转发回 GPT。为叙述流畅保留了"我"的视角，但读者应知：真正写代码、读日志、跑实验的是 AI。详见后文"真正的结构问题"一节。

## 现象

测试在 QEMU loongarch64（单核 / smp=1）下运行。60 秒后，QEMU 超时。内核日志的时间戳仅推进到 2.505 秒，之后没有任何新的内核输出。

扒开日志，三个数字跳了出来：

**138 → 95**。两次 fork 之间，父进程的 charge 计数从 138 掉到 95，丢失 43 个条目。中间只执行了 mremap 和 mprotect，理论上最多回收几个页面。43 个条目明显异常——**后来证明 138 和 95 来自不同进程的日志混在一起，并非同一个计数器下跌，见"死胡同 #2"。**

**"already Anon, syncing" 重复出现**。同一个虚拟地址 `0x601c2000`，`cow_file_write_to_anon` 被反复触发，每次都报告"这个页面已经是 Anon 了"。按照设计，MAP_PRIVATE 文件映射的首次 COW 写入把一页从 File 重分类为 Anon，之后就不应该再进入这个路径。重复进入意味着页面分类状态和 charge map 之间的映射可能已经漂移了。——**后来证实：父进程在 fork 前已对该地址做了 mprotect + COW 写，File→Anon 转换在父进程侧已完成。子进程继承时该页已是 Anon，再触发 COW 自然报告 "already Anon"。内核行为完全正确，属于正常语义，与 bug 无关。**

**内核静默挂起**。最后一条有意义的内核活动是 `handle_cow_fault(VA:0x3ffffe000, count=2)`，属于子进程的栈页。之后没有 panic，没有 oops，没有任何用户态输出。测试二进制在走到任何断言之前就死了。

三个现象看起来指向同一个方向：RSS accounting 的 charge map 存在一致性问题。charge 泄漏导致 138→95；分类漂移导致重复 "already Anon"；最终在某个 COW 路径上发生下溢 panic，只是 panic 信息没来得及输出到串口。

这个判断不是随意做出的。它同时解释了三个异常。它指向了内核中最复杂的那部分代码——我过去一周刚写好的 MemoryAccounting。它和我对"内核 bug 应该长什么样"的直觉完全吻合。

问题是，在这个判断里，有一个前提被跳过了：**bug 真的存在**。


<a id="dead-end-1"></a>

## 死胡同 #1：死锁——但内核根本没死

我的第一个调查方向是"挂在哪里"。日志显示 fork #3 的 PRE-FORK 出现了，但 POST-CLONE 永远没有出现。这意味着 `clone_map` 在遍历 VMA 的过程中卡住了。

DeepSeek 给出了一个自然的解释：SMP 下 FRAME_TABLE 锁与 AddrSpace Mutex 发生了 AB-BA 死锁。这是教科书级别的多核 bug 模式。我接受了这个方向，于是 DeepSeek 开始阅读 `clone_map` 的完整实现，梳理 FRAME_TABLE 的锁获取序列，设想了一个复杂的场景：CPU A 在 `clone_map` 里持有了 AddrSpace Mutex 然后等待 FRAME_TABLE；CPU B 在 page fault 路径里持有了 FRAME_TABLE 然后等待 AddrSpace Mutex。

为了验证，我在 `clone_map` 的每个关键步骤加了 `CLONE-PROGRESS` 日志，重新编译，重新跑 Docker，重新等待 QEMU 启动。

CLONE-PROGRESS 的输出告诉我：**clone_map 每一行都在正常执行。** 所有 VMA 都被遍历了，所有 frame 都被复制了。没有阻塞。没有死锁。

事后回头看，AB-BA 死锁虽然在 smp=1+sched-rr（抢占）下理论可能——通过抢占而非同时持锁实现——但概率极低。真正的问题不是死锁，是串口太慢：每个 `[RSS_VERIFY]` 日志行约 120 字节，几千行日志 × 每行 ~0.7ms 的串口传输时间 = 内核时间 2.5 秒的日志量需要 60+ 秒 wall clock 才能输出完。"POST-CLONE 没有出现"不是没执行，是没有传完。

教训 #1：在 QEMU 内核调试中，日志中没有出现 ≠ 代码没有执行。串口是低速串行链路，日志生成速度远超传输速度时会排队积压——日志没打印不代表代码没执行。


<a id="dead-end-2"></a>

## 死胡同 #2：138→95 的「charge 泄漏」

解决了超时（timeout 升到 120s）后，测试终于输出了真正的失败信息：

```
FAIL: child COW write increases RssAnon
parent_rss_anon=92 KB  (= 23 pages)
child_rss_anon=92 KB   (= 23 pages)
expected=96 KB          (= 24 pages)
```

子进程执行 COW write 之后，RssAnon 应该是 24 页（fork 继承 23 页 + COW 贡献 1 页），但实际只有 23 页。

与这个失败同时存在的，是 138→95 的 charge 下降。两条线索指向同一个假设：**charge map 在泄漏**。`clone_map` 复制 charge 时漏掉了条目，或者 `reconcile_fork_charges` 算错了。

我开始验证 BTreeMap 的内部一致性。在 `record_charge` 里加了 duplicate 检测。在 `remove_charge` 里加了 missing key 检测。在 `snapshot_resident_charges` 里加了迭代前后长度一致的断言。添加了两个全局 AtomicU64 计数器：`RECORD_OK` 记录成功的 insert，`RECORD_DUP` 记录重复的 insert。

等待 Docker 构建。等待 QEMU 启动。等待日志流。

`RECORD_DUP = 0`**。** 在所有测试运行中，没有一次 duplicate insert。没有一次 missing key 的 remove。`mapped_pages == charge_entries` 在所有 fork 边界上成立。`len_ok` 为 true——BTreeMap 迭代期间从未被并发修改。

BTreeMap 是干净的。charge map 没有泄漏。那 138→95 是怎么回事？

后来我才明白：138 和 95 不是同一个东西的两次测量。它们来自不同的 pid，在不同的时间点。我的 AtomicU64 计数器混合了 init、shell、测试二进制、子进程——所有进程的 reconciliation 输出混在一起。138 是进程 A 在时间 Ta 的峰值，95 是进程 B 在时间 Tb 的状态。它们之间的差值不是"丢失"，是"不相干"。

教训 #2：没有 PID 上下文的全局计数器在多进程内核中是不可靠的诊断工具。138 和 95 看起来像是变量从高降到低，但它们根本就不是同一个变量的值。

## 死胡同 #3、#4、#5：sync 覆写、fork 继承、跨进程共享

这三个假设可以放在一起说，因为它们共享同一个模式：每一项都在内核里找到了完美的无罪证明，每一轮验证都让内核看起来更正确。

**sync_rss_atomics_from_charges 覆写假设**：这个函数有一个 load-compare-store 序列。如果 SMP 下 atomics 在比较和写入之间被另一个 CPU 修改了，正确的值可能被错误覆写。添加 `SYNC-OVERWRITE` 计数器后——**0**。在所有调用中，`atomics_before == snapshot == atomics_after`。sync 永远是一个 no-op。它是防御性代码，从不执行实际的覆写。——**事后回头看，系统只有一颗 CPU，"另一个 CPU 并发修改"这个前提本身就不存在。但在调查当时，这个事实被所有人遗忘了。**

**fork 继承假设**：怀疑 `clone_map` 复制 charge map 时遗漏了某个 VMA 或 VA。在 fork 路径添加了 `FORK-INHERIT` 日志后——**parent_anon = 22, child_anon = 22**。每个 fork 边界上继承完全正确。Parent 和 child 有独立的 `acct_ptr`（指针不同），不是共享。

**跨进程 BTreeMap 共享假设**：怀疑父进程和子进程共享了同一个 BTreeMap 对象（通过 Arc），导致子进程的 COW 操作修改了父进程的统计。验证了每个 pid 的 `acct_ptr` 值——全部唯一。每个进程有独立的 MemoryAccounting 对象。

三连证伪。内核在我面前一寸一寸地被清洗干净。BTreeMap 正确，atomics 正确，fork 正确，隔离正确。到这个时候，我已经花了大约 8 个小时，逐一排除了内核 accounting 的每一个可能出错的路径（全程至定位根因，DeepSeek 平台统计累计消耗约 2.6 亿 token，含输入上下文与模型输出，主要来自长日志反复填入上下文窗口）。

然后我做了那个致命的综合：**内核 accounting 全部正确 + 23→22 的 drop 真实存在 = 一定有一个尚未被发现的内核机制在删 charge。** 这个推理在逻辑上完全自洽——如果 BTreeMap 的减少不是因为 sync/fork/sharing 的 bug，那就必是因为某个更隐蔽的、不在已验证路径上的内核操作。

它唯一的缺陷是前提错了。

## 转折点：承认 23→22 是真实的

在所有会计模块都被验证正确之后，只剩下一个事实无法绕开：**在 T1（**`/proc/self/status` **读取）和 T2（fork）之间，父进程 BTreeMap 的 anon 数确实从 23 变成了 22。**

我在 `stats.rs` 里加了两个日志点。`SNAPSHOT-T1` 在 `ProcessMemStats::collect()` 被 `/proc` 读取触发时打印。`SNAPSHOT-T2` 在 fork 路径中同样打印。两个日志点都输出了 pid、acct_ptr、anon count、file count、纳秒级时间戳。

日志回来了：

```
[SNAPSHOT-T1] pid=26 acct_ptr=0x9000000001b92070 anon=23 file=74 ts=2995115000
[SNAPSHOT-T2] pid=26 acct_ptr=0x9000000001b92070 anon=22 file=75 ts=3180000000
```

同一个 `acct_ptr`。同一个 `snapshot_resident_charges()` 调用。两次读取之间隔了 185ms。BTreeMap 里的 anon count 确实少了一页。不是统计噪音，不是 SMP 幽灵读，是 BTreeMap 里的一个条目被删除了。

两次读之间，用户态只做了 `pipe()`——两个同步的系统调用。不分配内存，不释放内存，不改变任何 VMA。

BTreeMap 是如何在被单线程持有 Mutex、没有任何系统调用的情况下发生变化的？这成了整个调查的核心矛盾。


<a id="dead-end-6"></a>

## 死胡同 #6：延迟清理——最长的死路

在排除了所有"在同一个时间点"的解释后，唯一剩下的可能是"时间"。BTreeMap 的修改不是发生在 T1-T2 这个窗口内的，而是**从更早的时间点延迟到这个窗口才被看到的**。

最自然的候选：test #4（`test_map_private_file_write_first_fault`）在 VA:0xf1000 做了 `munmap`。这是一个 Anon charge——munmap 会调用 `remove_charge(Anon)`，从 BTreeMap 里删除一条 Anon 条目。如果这个删除没有被及时"提交"（SMP 下 cache line 没有被 invalidate），test #6 的 `/proc` 读取（运行在另一个 CPU 上）就可能看到旧状态——BTreeMap 里还有这条 Anon charge。

这个假设在逻辑上无比自洽。它解释了为什么单线程、无系统调用的 T1-T2 窗口里 BTreeMap 会变化——因为修改不是在这 185ms 内发生的，而是之前发生的，只是对当前 CPU 的可见性延迟了。

我把这个假设输入了这条调查流水线。回头看，得到的回应几乎每句话都让我更确信这个方向：

> "这既不是测试逻辑错误，也不是 MemoryAccounting 的计算错误。是 `munmap` 的 BTreeMap 修改与后续 `/proc` 读取之间的时序问题。"

这个结论听起来专业、严谨，区分了"测试设计"、"数据结构正确性"、"执行时序"三个维度。它精准地排除了我内心的主要怀疑对象——测试代码。几百行 C 的测试怎么可能比几万行 Rust 的内核更可疑？

为验证这个假说，执行了两个实验。和整场调查中的每一轮一样——GPT 提方案，我转发，DeepSeek（Claude CLI）实施。

第一个实验：在 `remove_charge` 里添加 `caller` 参数，在 `sys_munmap` 调用处传入 `caller=sys_munmap`。同时在 `sys_mmap`、`sys_munmap`、`sys_fork` 的入口加了 tracepoint，打印 pid、VA、纳秒级时间戳。

第二个实验：在测试 C 代码里添加了 `clock_gettime` 包裹每个 `read_self_status_kb` 调用，试图将内核时间戳和用户态时间戳对齐。

日志回来了。VA:0xf1000 的 `remove_charge` 的时间戳是 ~2.17 秒。T1 是 ~2.69 秒。T2 是 ~3.18 秒。

**VA:0xf1000 在 test #4 期间就已经被清理了。** 它不在 T1 和 T2 之间。延迟清理假设被时间戳证伪。

那个在 T1-T2 之间消失的 Anon 条目，不是 VA:0xf1000。那它是谁？


<a id="va-16000"></a>

## VA:0x16000

在 test #6（失败的那个测试）执行期间，日志里出现了另外一个 `remove_charge` 事件：

```
[CHARGE-MUTATION] ts=32501686752 pid=23 op=remove va=VA:0x16000 old_kind=Anon caller=sys_munmap
```

VA:0x16000。不是任何已知测试 VMA 的地址。

我回去搜了 SYSCALL-MMAP 日志。同一时间窗口内，这个地址的 mmap 记录：

```
[SYSCALL-MMAP] ts=2697269616 pid=23 va=VA:0x16000 len=0x1000 file=false flags=READ|WRITE|USER
```

`file=false`。这是一个**匿名映射**，不是测试的 MAP_PRIVATE 文件映射（VA:0x15000, `file=true`）。它在 test #6 的 T1 测量之前被创建，在 T1 之后被销毁，生命周期约 1.73 ms。

完整的因果链：

```
2696539024 ns: 测试的 mmap(VA:0x15000, file=true)          ← 文件映射
2696854544 ns: record_charge File at VA:0x15000             ← 缺页填充
2697269616 ns: SYSCALL-MMAP va=0x16000 file=false           ← fopen buffer！
2697432688 ns: record_charge Anon at VA:0x16000             ← anon: 22→23
2697500000 ns: SNAPSHOT-T1 anon=23                          ← T1 快照（污染值）
2699002320 ns: SYSCALL-MUNMAP va=0x16000                    ← fclose 释放
2699263520 ns: remove_charge Anon at VA:0x16000             ← anon: 23→22
```

T1 快照时 anon=23，VA:0x16000 的 Anon page 仍然存在；约 1.7 ms 后它被 munmap。快照捕获到了它，fork 没有。

**一个讽刺的细节**：捕获这条时间线的 `SYSCALL-MMAP`/`SYSCALL-MUNMAP` tracepoint，原本是为了追 VA:0xf1000 的延迟清理——一个错误方向——而加上去的。为错误目的埋的桩，意外捕获了真正的凶手。

> 说明：这条时间线取自 loongarch 插桩运行（基线 22→23，92 KB / 23 页）。同一 bug 在 aarch64 上也复现——基线 18→19，delta 同为 +1/−1。绝对基线随架构不同，但 ±1 的 delta 完全一致——测量自身扰动 RSS 是跨架构可复现的。

这个匿名映射不是测试创建的。它是 `fopen("/proc/self/status", "r")` 创建的。

验证链路如下——事后在 Linux 上用同版本 musl 重放，strace 直接捕获到了完全一致的行为：`fopen` → `mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS)`，`fclose` → `munmap(4096)`。两个独立测量确认了分配量级：LD_PRELOAD 劫持 `malloc` 测得 musl `fopen` 的单次总分配请求为 1264 字节（= FILE 结构体 + UNGET + BUFSIZ ≈ 232 + 8 + 1024）；其中纯 I/O 缓冲区请求为 1181 字节，按页粒度对齐到 4096——两种口径一致指向同一结论：fopen 会触发一次匿名 mmap。StarryOS 侧有对应的 QEMU 日志：`SYSCALL-MMAP va=0x16000 len=0x1000 file=false flags=READ|WRITE|USER`——匿名映射、与 `/proc` 读取完全对时。内核代码路径已验证：`cow.rs:123-131` 对 `file=None` 的映射将新页标记为 `RssKind::Anon`，`accounting.rs:107-108` 将其计入 `rss_anon` 计数器。

所以真相是这样的：

**污染发生在父进程的测量上，不在子进程。**

父进程调用 `read_self_status_kb("RssAnon")` 读取自己的 `/proc/self/status`。`fopen` 在 musl libc 上触发了一次匿名 `mmap(4096, MAP_ANONYMOUS)`——文件结构体、unget 缓冲、读写缓冲共 1264 字节，按页粒度向上取整到 4096。`fopen` 第一次写入 FILE 结构体时触发 page fault，StarryOS 内核将该匿名页计入 RssAnon。然后 `fgets` 解析出 `"RssAnon: XX kB"`——**父进程读到的值比自己的真实 RSS 多了一页**。那页正是 `fopen` 自己的 FILE buffer。

`fclose` 关闭文件，`munmap` 释放了这个匿名页面，随后 `free` 回收 FILE 结构体。父进程的 RSS 回到了真实值（22 页）。

然后 `fork`。此时 fopen buffer 早已释放，子进程继承的是父进程的真实值——22 页，干净值。子进程 COW 写，anon 从 22 升到 23。子进程读自己的 `/proc/self/status`——读到 23，这是正确的。

断言检查 `child >= parent + 1`。parent 用的是 T1 的污染值 23（真实 22 + fopen 的 1），child 是 23（真实 22 + COW 的 1）。23 >= 23+1 → FAIL。

**父进程被 fopen 多计了一页，子进程是干净的。污染在父不在子。**

bug 不存在于内核。它在测试代码里。在 `fopen` 那一行。

`fopen` → `open`。这是修复。


<a id="test-acquittal"></a>

## 测试被两次正式提审，两次无罪释放

在调查过程中，"会不会是测试的问题"这个问题被两次摆上桌面——但摆上桌面的不是我，是 AI 自己。GPT 在结构化调查中把它列为一个分类（Q5："这是测试逻辑问题还是内核问题？"），DeepSeek 收到了问题、给出了分析、判了无罪。我在两个 AI 之间做信使——把 GPT 的 Q5 粘贴到 CLI，把 DeepSeek 的判决粘贴回 ChatGPT。

两次，DeepSeek 都给出了清晰的、逻辑严谨的无罪判决。GPT 在第二轮的延迟清理调查中起了推动作用——要求加 caller= 追踪、做 A/B/C 分类——但具体的无罪判词来自 CLI 侧。

第一次，DS 在保存 checkpoint 前做的调查总结：

> "这既不是测试逻辑错误，也不是 MemoryAccounting 的计算错误。是 `munmap` 的 BTreeMap 修改与后续 `/proc` 读取之间的时序问题。"

注意这句话的结构：它否定"测试逻辑错误"和"会计计算错误"，但把问题归因于第三种可能——"时序问题"。这个分类看似精确，实则漏掉了唯一真正的答案：**测量工具的副作用**。`fopen` 的 FILE buffer 污染 RSS——这不属于"测试逻辑"（测试的文件读取逻辑是对的），也不属于"会计计算"（内核的计算是对的），也根本不是"时序问题"（fopen 的副作用在任何时序下都存在）。

这段分析用了一个看似穷尽的三分类，把真凶排除在了分类之外。

第二次，在调查 "延迟清理" 假设时：

> "不是测试设计问题(A)：测试逻辑正确——读 /proc/self/status → fork → 子进程 COW 写 → 读子进程状态。测试没有并发、没有竞态假设。C 测试代码在两个 pipe 之间没有触发任何 unmap 的操作。"

> "不是 MemoryAccounting 问题(B)：BTreeMap 本身的数据结构操作（insert/remove/iterate）是正确的……问题在于'计数发生在错误的时间'……"

> "是进程生命周期问题(C)：AddrSpace::clear() 的执行时机与进程调度解耦……"

分析精确、推理完整、分类清晰。完全错误。

两次判决共享一个前提：内核的某个操作（munmap、AddrSpace 清理）被延迟到了 T1-T2 窗口内执行。两套分析都从未考虑过"读取操作本身可能分配内存"——不是因为逻辑上有漏洞，而是因为测量工具（`fopen`）在前提检查的范围之外。

测试被预先赦免了。

但更值得追问的是另一个问题：**真凶的踪迹在日志里出现过吗？**

答案是：出现过，至少三次。

第一次，在调查刚开始时（第 3 轮）。DS 遍历所有 VMA 寻找不变量被破坏的证据。VA:0x16000 的匿名 mmap 日志就在输出中——`file=false`，发生时间恰好在 test #6 的 `/proc` 读取窗口内。但在 138→95、CPU26/27、"already Anon" 三重噪音下，一个看起来像 libc 内部分配的匿名映射，根本不值得提。

第二次，在"震荡"发现时（第 18 轮）。同一个进程，两次 `/proc` 读取之间，RssAnon 在 23 和 22 之间来回。DS 已经定位到 `/proc` 读取和 fork 之间 BTreeMap 发生了变化——**如果能追问一句"谁在两次读之间分配了内存"，fopen 就在这里结束了。** 但当时的问题是"哪个内核模块删了一页"，不是"用户态什么东西加了一页又删了"。

第三次，在第 23 轮。DS 从日志中发现同一个 VA 在 1ms 内先被 mmap 再被 munmap，提议添加调用栈追踪。这个 VA 就是 0x16000。但注意力立刻被"进程退出清理"和"共享 MemAccounting 指针"两个更复杂的内核解释抢走了。

三次，fopen 的脚印就在日志里。三次，调查从它旁边走过。

这不是视力问题。是**问题框架决定了什么算证据**。当框架是"内核有 bug"时，libc 的内存分配行为不在证据空间内——它属于"环境噪声"。要看到 fopen，需要的不是更多日志，而是把框架从"内核哪里错了"切换到"谁改了这个数字"。


<a id="structure-problem"></a>

## 真正的结构问题：不是"AI 不够好"，是"人类是信使"

回看这条流水线，有一个比"验证为什么还是验证是否"更深的结构问题：**我在这次调查中到底做了什么？**

期末考周。我把测试交给 Cursor 生成。测试失败后，我把日志喂给 Claude CLI（DeepSeek 模型）。DeepSeek 输出分析，我复制粘贴给 ChatGPT 网页版。ChatGPT 输出下一段分析，我复制粘贴回 DeepSeek。我在 59 轮对话中的实际角色是**消息队列**——接收一个模型的输出，转发给另一个模型。

这个结构有三个后果，每一个都比"模型犯了错误"更严重。

**第一，错误在转发中固化为"事实"。** DeepSeek 在 Round 3 说"CPU 26 和 CPU 27 都参与过该 fault"。我把它原样写进了 Round 4 的 prompt，标注为"已确认事实 #3"。ChatGPT 基于这个"事实"继续推演。实际上日志格式是 `{cpu_id}:{tid}`，DeepSeek 把 task_id 26 误读成了 CPU 26——在 smp=1 的系统上，只有 CPU 0。但我不读代码，我只看模型的文本输出。模型说 CPU26，我写 CPU26。**错误不是模型独有的——它在我这一步被"确认"成了下一轮的输入，而我对此没有任何检验。**

**第二，不该存在的假说活了几个小时。** AB-BA 死锁在 sched-rr 抢占下单核理论上可能（通过抢占而非同时持锁），但概率极低，且真正的问题是串口瓶颈。而 SMP atomics 覆写、cache coherence 这些假设在单核上就完全不可能——另一个 CPU 根本不存在。QEMU 配置的目录名里就写着 "smp1"——单核。整个调查期间**没有人在看代码和配置**。模型在读代码和日志，人在读模型的文本。smp=1 这件事，从模型文本里是看不到的。不是因为它在某个深奥的配置文件里——目录名就在那里——而是因为人根本没去看，模型也没把它放进推理的前置约束里。

这里还有一个更深的规律：**LLM 在长上下文推理时会丢失"明显的事实"。** smp=1 在对话开头提过一次，300 轮之后它不再是 active context。AI 不会像人那样记住"我们只有一个核"——除非被反复提醒。这暗示 AI 辅助调试可能需要一个自动维护的"已知约束列表"，每次推理前做一致性检查。

**第三，"人"这个环节没有提供任何新增信息。** 理想的 AI 辅助调试：人理解系统、人提出假设、AI 执行验证、人判断结果。实际发生的：AI 提假设、AI 执行验证、AI 分析结果、人做 Ctrl+C / Ctrl+V。两个模型共享同一个未被审查的前提，而中间那个人——唯一有能力跳出文本、质疑前提的实体——因为考试没时间读代码、因为觉得"简单功能很快就修完"、因为没有在模型输出之外独立查证任何一个断言——实际上没有在流水线中增加任何新信息。**AI 擅长枚举可能性，人擅长排除不可能性。** 但在这场调查里，枚举在狂奔，排除从未发生。

公平地说，我并不是完全没有动手。GPT 建议"先在 Linux 上对照"时，我在 WSL 上编译运行了测例，看到 `FAIL: touch one page increases RSS`。但这次唯一的独立验证，目的不是"查证 DeepSeek 或 GPT 的断言是否成立"，而是"验证 Linux 表现是否和 StarryOS 一样"。验证了错误的问题。它没有帮我质疑前提——反而让我觉得"连 Linux 也跑不过，测试本身就很难通过"，把测试推得更远了。

这不是 AI 能力的问题。DeepSeek 和 ChatGPT 在整个过程中表现得足够好——它们读代码、写 tracepoint、分析日志、设计实验。问题在于**它们共享了错误的前提**，而**唯一能打破这个前提闭环的人，没有在闭环里。**


<a id="premise"></a>

## 这个前提为什么能活下来

事后看，有四个机制让"bug 存在于内核"这个前提始终免受检验。

**第一，测试是参考 Linux/kselftest 写出来的。** 它在诞生时就带有权威感——它模仿了 Linux kselftest 的测试结构（`TEST PASSED`/`FAIL:` 格式、`/proc` 字段解析、fork 后的父子验证），模仿了 psutil/top 的使用模式。当一个测试在"参考 Linux 主线 + kselftest + psutil"的指令下生成时，它的可信度被原创者的权威置换了。

但它漏掉了一条关键语义：在被测进程内部用 `fopen` 读 `/proc/self/status`，读取操作本身会分配内存，内存会被计入 RSS。在我测试环境中的 glibc 上，同样的 `fopen` 没有复现 +1 页的现象（glibc 的 `fopen` 走 `brk` 扩堆、复用已有页面，不一定触发新的匿名 `mmap`），但 StarryOS + musl 上每次 `fopen` 都独立 `mmap` 一页匿名内存。测试复制了 Linux 测试的"形"，没有捕捉到这条 musl 特有的副作用。

**第二，信任排序不依据证据，而依据复杂度。** 内核是几万行 Rust、跨越 MM、VMA、COW、fork、SMP 多个子系统、我刚写了不到一周的代码。测试是几百行 C、逻辑简单、单文件、几十个函数调用。在"哪个更可能出错"的直觉排序里，内核排在测试前面，差距大概一个数量级——不是因为任何证据表明内核有 bug，而是因为内核有"看起来能出错"的地方更多。

**第三，信任链的起点从未进入验证范围。** 我验证了 BTreeMap 的 insert/remove consistency。验证了 atomics 与 snapshot 的一致性。验证了 fork 继承的逐 VA 正确性。验证了 COW File→Anon 重分类的正确性。甚至验证了 SMP 下 Mutex Release-Acquire 的内存可见性保证。但我从未验证"`read_self_status_kb` 的返回值是否等于内核实际状态"——因为它被当成了测量基准，而基准不在验证范围内。我不是在验证"bug 是否存在"，我是在验证"为什么 bug 存在"。

**第四，伪独立性。** 测试被两个模型分别"提审"了至少两次，每次都给出专业的无罪判决，每次都因不同的"更合理的内核解释"而放过了测试。两次审查看似独立——不同时间、不同上下文、不同分析路径——实际上共享了同一个未被质疑的前提。一个在独立审查中幸存了两次的测试，看起来比单次审查更可信——但两次审查都来自同一前提的衍生，而非真正的独立验证。

每一次验证内核模块的正确性，都让我在潜意识里更确信"一定有一个内核 bug 我没找到"，而不是"也许内核没问题"。

正确的结果，错误的方向。


<a id="interlude"></a>

## 插曲：这个 bug 是三个 AI 接力写出来的

回头数一数，这次 bug 其实有三个 AI 参与了创作。

**Cursor autocomplete** 随手写了 `fopen`——不知道自己在写什么，不知道 musl 和 glibc 的区别，不知道 `fopen` 会在目标系统上涨 RSS。它只是在上下文中看到了 `/proc` 文件路径，补全了一个似乎合理的文件读取方式。

**DeepSeek（Claude CLI）** 写了测例的其余部分，没有质疑 `fopen`。但在调试阶段，它自己从 tracepoint 的输出中——`file=false`、匿名映射、时间戳跨 T1——推断出了 fopen 的缓冲区行为。同一个事实（fopen 会 mmap），写代码时没看到，调 bug 时才看到。同一个模型，两套认知模式，中间没有打通。

**GPT（ChatGPT Web）** 设计了 Audit Mode 的 Q1-Q5 框架，强制执行"no guessing"纪律。它负责调查的结构化组织——分类假设、排除已证伪路径、要求 caller= 追踪——但直到 R45 才从 DS 的输出中间接看到 fopen。它擅长组织调查，但不擅长发现具体事实。

没有一个 AI 是全知全能的。每个都在自己的局部上下文里盲写。Cursor 不知道 musl，DS 在写代码时没关联到 fopen 的副作用，GPT 从未直接读日志。三个 AI 的输出像三块拼图，拼在一起才构成完整的 bug——但拼图的人不是 AI，是那个考试周里忙着复制粘贴的人。


<a id="fix"></a>

## 修复

```diff
- FILE *fp = fopen(path, "r");
+ int fd = open(path, O_RDONLY);
```

五个函数，每个三行改动。重新编译，重新跑测试。通过。

不是 charge 泄漏，不是死锁，不是 atomics 覆写，不是 fork 继承 bug，不是延迟清理，不是 cache coherence。六个被逐一证伪的假说。是一行 `fopen`。


<a id="what-if"></a>

## 如果当时……

如果测例在每次 `fopen` 读 `/proc/self/status` 之后，立刻用 `open`/`read`/`close`（不走 `fopen`，不走 `FILE*`）再做一次独立的验证读取，两次读的 RSS 值理应完全一致——不一致就是测量扰动。这项检查加在 `read_self_status_kb` 内部，一行代码的事。

如果在 fork 之前，父进程把此刻的 RSS 打印到 stderr，和 fork 后子进程继承的值对比——fopen buffer 已释放，父子基线应该相同。不同就是有问题。

如果在每次 `/proc` 读取前后各打一个 RSS snapshot，diff 非零就意味着"读操作本身分配或释放了内存"——不需要读内核代码，不需要 tracepoint，不需要 2.6 亿 token。

**不是事后诸葛亮。** 这三项检查加起来不到 20 行 C 代码，属于任何一个测量系统的基础自检——在物理实验里叫"标定"，在软件测试里叫"sanity check"。Cursor 不会主动加这些，因为 AI 默认前面的步骤正确、后面的 FAIL 才是 bug。但实际发生的是：错误在第一步就埋下了，后面的断言全在错误基础上推演，不可能对。


<a id="ending"></a>

## 结尾

这次调查从第一天下午持续到第二天下午，跨了约 22 个小时——其中睡了一觉，AI 没睡。真正在"调试"的时间——读代码、查 man page、做实验、对比结果——大概不到两小时。其余时间是在等 Docker 构建、等 QEMU 启动、等模型的上下文窗口装下日志、等一个人的分析复制粘贴到另一个模型。

我并不是在指责模型犯了错误。问题不在这里。问题是：**当人类在一个调试闭环中唯一的贡献是复制粘贴时，错误不会被纠正，只会被转发。**

功能修好了。三门早考的科目也考完了。但这条流水线上的每一个"为什么对"和"为什么错"，到底有多少是我亲手验证过的？

在调查全程中：零个。所有代码阅读、日志分析、假说提出、实验设计、结论判断，均由 AI 完成。我在中间做复制粘贴，没有独立验证过任何一个断言。

调查结束后，复盘时：一个——`fopen` 换成 `open`，12 个测试全绿。而且即使这个，也是 DeepSeek 写完修复、跑完测试、报告"通过了"之后，我在事后才确认的。修复代码本身不是我写的。

RSS 从来没有说谎。它如实记录了包括测量行为在内的所有内存变动——`fopen` 的 FILE buffer、COW 的 File→Anon 重分类、子进程的栈页缺页。说谎的是我们对"测量操作无副作用"的默认假设，以及那个从未被验证的前提："bug 一定在内核里"。

但这场调查留下的教训不止一个：

1. **约束前置。** 设计函数和测例时，把不合法路径判死。每个步骤自证。断言不能只验证结果对不对，还得验证中间状态是否合法。
2. **人审 AI，审的是不可能性。** AI 提出假说时，先问"这个假说违反了哪些已知约束？"而不是"这个假说有没有证据？"

   具体到这次调查：
   - 死锁假说（死胡同 #1）→ 问：smp 配置是几？→ **smp=1，只有一颗 CPU。** 抢占下单核死锁理论上可能，但概率极低。先查串口，再查死锁。
   - 跨进程共享假说（死胡同 #5）→ 问：fork 之后父子共享地址空间吗？→ **不共享。** 子进程有独立页表，子进程的 BTreeMap 操作不可能污染父进程的统计。VA 隔离直接判死。
   - 延迟清理假说（死胡同 #6）→ 问：单核上 cache coherence 延迟存在吗？→ **不存在。** 同一个 CPU 上的写操作立即对自己可见。没有第二个 CPU 可以"看到旧值"。

   AI 擅长枚举可能性，人擅长排除不可能性。smp=1、fork 语义、VA 隔离——这些不是细节，是判死假说的快刀。三刀下去，六条死胡同至少砍掉四条。但整个调查期间，没有人在 AI 提出假说时问过这些问题。
3. **先查 man，再读代码。** 查规范回答"应该怎样"，读代码回答"实际怎样"。AI 倾向于读代码（因为代码在 repo 里可以直接 grep），但规范往往比代码更快地回答"应该是怎样"这个问题。`fopen` 在 musl 上的实现——`mmap` 分配 FILE 缓冲区——一行 `strace` 就能看到。
4. **debug 分支写结论，不写修改。** commit message 记录证伪/证实了什么，不是改了什么。每个阶段到达的结论是唯一永久有价值的产出。可以强化：每次 commit 加 `Confirms:` 或 `Refutes:` 标记。
5. **±1 先怀疑 libc。** RSS 差一页——动态链接的缺页、fopen 的 FILE buffer、malloc 的 brk。这些都比内核 bug 更常见。4KB 的嫌疑犯列表上，libc 排第一。
6. **关注 AI 的上下文衰减。** LLM 在长上下文推理时会丢失"明显的事实"。smp=1 在对话开头提过一次，300 轮之后它不再被考虑。AI 辅助调试需要一个自动维护的"已知约束列表"，每次推理前做一致性检查——而不是依赖模型自己记住。
